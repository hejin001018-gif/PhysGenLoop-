## 一、问题根因（实施前确认）

| 编号 | 问题 | 代码位置 |
|---|---|---|
| P0-1 | `CriticReport` 跨进程反序列化丢 violations | `schemas.py` 无 `from_dict`；`sam2_vlm_critic.py` 调用不存在方法被 except 吞掉 |
| P0-2 | mask 证据链断裂 | `eval_step.py` 写了 mask_uri，但父进程反序列化把 violations 清空 |
| P0-3 | ProPainter 后端缺失 | `/root/ProPainter` 不存在，executor 注册了但必然失败 |
| P0-4 | 四动作只有 local_editing 走 executor | `controller.py` prompt/global 直接改 prompt，不经 executor |
| P0-5 | Prompt Executor 二次调用 Policy | `PromptRepairExecutor` 内再调 `repairer.repair()` |
| P0-3m | proxy memory 格式不兼容 | checkpoint 是 `LearningTargetV1`，loader 期望 `RepairExample`，每样本报错吞掉 |
| P1 | 无 repair_trace / trial / memory_status / resource 审计 | 旧链路无对应产物 |

## 四、双卡策略与分辨率说明

**策略：双卡角色分工**
- GPU0：Wan2.2 生成（`WAN_GPU_ID=0` 隔离子进程）
- GPU1：Qwen3-VL vLLM 常驻（`CUDA_VISIBLE_DEVICES=1` 启动）
- 效果：旧链路每轮 vLLM 冷启动 ~40s × 30 次 = ~1200s 开销消除；两卡同时有负载

**分辨率：704×1280 → 480×832**
- 像素量降至原来 44%，理论生成 ~2.2× 提速
- 对齐 diffusers WanPipeline 的原生默认分辨率档
