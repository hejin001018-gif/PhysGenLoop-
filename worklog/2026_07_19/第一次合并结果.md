- hejin 的 `src/physgenloop/learning_repair/` 合入，旧 `learning_repair_pipeline` 改为 deprecated shim

### 2.2 集成层空缺（本次主要工作）

合并后代码在各自的边界内完整，但集成层是空的——组件之间没有人把它们连起来。发现并修复了以下七个问题：

---

#### 问题 1：没有顶层闭环入口

**现象**：没有任何脚本把 `LoopController + WanSubprocessGenerator + Sam2VlmSubprocessCritic + ActionValueRepairer` 组装在一起跑完整闭环。

**解决**：新建 `run_loop.py`，实现子进程 GPU 交接式闭环（Wan2.2 生成释放显存后 vLLM 评估循环），每次运行输出到带时间戳的独立子目录 `outputs/run_YYYYMMDD_HHMMSS/`。

---

#### 问题 2：`Sam2VlmSubprocessCritic` 与 `ActionValueRepairer` 状态漂移

**现象**：`Sam2VlmSubprocessCritic` 内部维护了 `_attempt_index` 和 `_previous_actions`，`ActionValueRepairer` 也维护同一套状态，两处各自递增，多轮修复时对不上，修复决策上下文错乱。

**根本原因**：Critic 承担了修复上下文管理职责，违反单一职责原则。

**解决**：从 `Sam2VlmSubprocessCritic` 完全移除状态字段，`eval_step.py` 只做评分不做修复决策，修复上下文由 `ActionValueRepairer` 唯一持有。

---

#### 问题 3：`eval_step.py` / `gen_step.py` 不存在

**现象**：`Sam2VlmSubprocessCritic` 和 `WanSubprocessGenerator` 调用的子进程脚本在代码仓库中不存在，运行时直接报错。

**解决**：
- 新建 `gen_step.py`：加载底层 `WanGenerator`，生成视频后写 JSON 退出
- 新建 `eval_step.py`：启动 SAM2+VLM Critic，打分后写 JSON 退出，不持有任何修复状态

---

#### 问题 4：`gen_step.py` 使用常驻封装而非底层推理，且缺少生成参数

**现象**：子进程是一次性的，但调用的是常驻式封装 `WanPhysicsGenerator`，语义矛盾；htz 需要的 `num_frames`、`height`、`width`、`fps`、`negative_prompt` 五个参数没有暴露，全部硬编码。

**解决**：`gen_step.py` 改用底层 `WanGenerator` 直接调用，新增五个 CLI 参数并设合理默认值（81帧、704x1280、24fps）；`WanSubprocessGenerator` 同步加上这五个参数并透传到子进程。

---

#### 问题 5：Executor 框架存在但全是存根，无法采集真实 Trial

**现象**：hejin 设计的四个 Executor 框架完整，但没有注入真实后端，训练数据采集无法进行，`deployment_ready=false`。

**解决**：
- 新建 `generators/wanphysics/local_editor.py`：`ProPainterLocalEditor`，接受 SAM2 mask 或 bounding box 构造 inpainting mask，调用 ProPainter 子进程完成局部修复，重新编码为 mp4
- 新建 `generators/wanphysics/executor_factory.py`：`build_executor_registry()` 工厂，将四个 Executor 全部注入真实后端

---

#### 问题 6：所有路径和参数硬编码

**现象**：`run_loop.py` 中 `CKPT_ROOT`、`PROMPT`、`MAX_ROUNDS` 等全是模块级常量，每次换 prompt 或 checkpoint 都要改源码，违背 hejin 路线方案中通过 `configs/` 统一管理的要求。

**解决**：新建 `configs/loop.yaml`，将所有运行时参数统一管理。`run_loop.py` 和 `batch_run_loop.py` 均从 yaml 读取默认值，CLI 参数优先级高于 yaml。

---

#### 问题 7：htz 的批量 prompt 流水线没有对接 `LoopController`

**现象**：htz 的 `batch_generate.sh` 调用旧路径，与新架构完全脱节，只是单纯生成视频，没有闭环逻辑和 Critic 评估。

**解决**：新建 `batch_run_loop.py`，遍历 `prompts/` 目录下所有 `.txt` 文件，每条 prompt 独立跑一次完整 `LoopController` 闭环，输出兼容 htz 预留的 `critic.json` 接口格式，所有 prompt 共享同一个 vLLM 实例。

---

#### 已确认无需修复的项目

| 项目 | 结论 |
|---|---|
| htz 的 Mac 端 SSH/rsync 脚本 | 代码已全部合并到服务器，不再需要远程连接，废弃 |

| `LearningRepairLoopRunner` 与 `LoopController` 职责重叠 | 明确分工：`LoopController` 负责生产推理，`LearningRepairLoopRunner` 负责训练数据采集，两者并存合理 |
| sy 的 Qwen3-VL LoRA 微调计划 | 计划阶段，依赖外部数据准备，本次不实施 |

## 三、最终系统架构

### 3.1 整体架构图

```
输入层

        |
        v
+--------------------------------------------------+
|  run_loop.py / batch_run_loop.py                 |
|  从 configs/loop.yaml 读取配置                    |
|  CLI 参数可覆盖 yaml 值                           |
+--------------------+-----------------------------+
                     |
                     v
+--------------------------------------------------+
|  LoopController  (src/physgenloop/controller.py) |
|  max_rounds 轮循环，acceptance_score 判断是否接受  |
|                                                  |
|  +-- WanSubprocessGenerator                      |
|  |     子进程推理，退出即释放显存                   |
|  |     gen_step.py -> WanGenerator (diffusers)   |
|  |     参数：num_frames/height/width/fps          |
|  |                                               |
|  +-- Sam2VlmSubprocessCritic                     |
|  |     自动管理 vLLM 生命周期                      |

|  |       Qwen3-VL-8B (vLLM) -> 物理规则打分       |
|  |     输出：CriticReport(physics_score,decision) |
|  |                                               |
|  +-- EvidenceAwareSelector                       |
|  |     从候选中选出物理分最高的                     |
|  |                                               |
|  +-- ActionValueRepairer                         |
|        repair-agent-v3.1-proxy checkpoint        |
|        根据 CriticReport 决策修复动作并改写 prompt |
+--------------------------------------------------+
                     |
                     v
+--------------------------------------------------+
|  outputs/run_YYYYMMDD_HHMMSS/                    |
|  +-- {candidate_id}/                             |

|  |   +-- prompt.txt                              |
|  |   +-- metadata.json                           |
|  +-- critic.json    <- 兼容 htz 接口              |
|  +-- loop_result.json                            |
|  +-- vllm.log                                    |
+--------------------------------------------------+
```

### 3.2 GPU 交接机制

40GB A100 无法同时常驻 Wan2.2（约 32GB）和 vLLM（约 17GB），通过子进程交接解决：

```
主进程 (run_loop.py)
  |
  +-- 启动 gen_step.py 子进程
  |     加载 Wan2.2 -> 生成视频 -> 写 JSON -> 退出（显存释放）
  |
  +-- 启动 vLLM 服务（自动检测 GPU 是否可用）
  |     等待 /health 就绪
  |
  +-- 启动 eval_step.py 子进程
  |     加载 SAM2 -> 跟踪 -> 调 Qwen3-VL -> 写 JSON -> 退出
  |
  +-- 主进程读取 JSON，更新 LoopController 状态，决定是否继续循环
  |
  +-- 结束后 critic.shutdown() 停止 vLLM
```

### 3.4 关键组件接口契约

**`LoopController.run(prompt)` 返回 `LoopResult`**
```python
LoopResult(
    best=CandidateEvaluation(candidate, report),
    history=[LoopRound(...)],
    stop_reason="accepted" | "max_rounds",
    resolved_plan=PhysicsPlan,
)
```

**`CriticReport` 关键字段（`Sam2VlmSubprocessCritic.evaluate()` 返回）**
```python
CriticReport(
    physics_score: float,   # 0.0-1.0
    decision: str,          # "physical" | "violation"
    confidence: float,
    coverage: float,
)
```

**`configs/loop.yaml` 可配置项**
```yaml
loop:
  prompt: "..."       # 默认 prompt
  max_rounds: 2       # 最大修复轮数
  acceptance_score: 0.8

generator:
  num_frames: 81      # 4n+1 格式
  height: 704
  width: 1280
  fps: 24

vllm:
  gpu_util: 0.85
  max_model_len: 16384
```
