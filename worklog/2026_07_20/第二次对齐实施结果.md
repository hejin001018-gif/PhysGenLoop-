## 一、sy-check.md 对齐结果

### P0-1 ✅ 生产入口接入四动作 ExecutorRegistry

**问题：** `run_loop.py` + `LoopController` 只调用 `ActionValueRepairer.repair()` 改写 prompt，四个动作标签全被压成"改 prompt 后重新生成"。

**修改：**
- `src/physgenloop/controller.py`：新增 `executor_registry` 可选参数；在每轮未 accepted 后优先调 `repair_with_decision()`，按 action_str 分发：
  - `reject` → 提前终止，`stop_reason="rejected"`
  - `local_editing` → 通过 `executor_registry.get("local_editing")` 执行 ProPainter，失败降级
  - `global_regeneration` → 重置为原始 prompt
  - `prompt_repair` → 使用修复后 prompt
  - 若 repairer 无 `repair_with_decision`，退化到原有 `repair()` 行为（向后兼容）
- `agents/wanphysics/run_loop.py`：构造 `ExecutorRegistry` 并注入 `LoopController`；新增 `--no-executor-registry` 调试开关

### P0-2 ✅ RejectExecutor 注入 selector + docstring 修正

**问题：** `executor_factory.py` 调用 `RejectExecutor()` 缺少必填的 keyword-only `selector`；docstring 使用错误参数名 `registry=`。

**修改：**
- `generators/wanphysics/executor_factory.py`：`RejectExecutor(selector=EvidenceAwareSelector())`；docstring 改为 `executors=`

### P0-3 ✅ Local Editor 消费正确字段

**问题：** `local_editor.py` 访问不存在的 `target.mask_path` 和 `target.bbox`，标准 `LocalEditTarget` 不含这两个字段。

**修改：**
- `generators/wanphysics/local_editor.py`：改用 `target.mask_uri`（SAM2 输出的 mask 图片路径）和 `target.critical_frames`（需修复的帧索引集合）；无 mask_uri 时退化为全帧白色 mask

### P0-4 ✅ CriticReport 完整反序列化（violations / evidence_bundles）

**问题：** `sam2_vlm_critic.py` 的 `_report_from_dict()` 只恢复 6 个基础字段，丢弃 violations 和 evidence_bundles，导致 RepairPolicy 特征提取退化为默认值。

**修改：**
- `generators/wanphysics/sam2_vlm_critic.py`：优先调 `CriticReport.from_dict()`（如存在）；降级时手动重建 `violations`（`Violation.from_dict`）和 `evidence_bundles`（`EvidenceBundle.from_dict`），异常时静默为空元组

### P0-5 ✅ PhysicsPlan 跨子进程传递

**问题：** `LoopController` 解析出的 `resolved_plan` 没有传进 eval_step.py 子进程，后者固定使用空 `PhysicsPlan()`。

**修改：**
- `generators/wanphysics/sam2_vlm_critic.py`：`evaluate()` 把 `physics_plan.to_dict()` 序列化写入 candidate JSON
- `agents/wanphysics/eval_step.py`：从 candidate JSON 读取 `physics_plan` 并用 `PhysicsPlan.from_dict()` 恢复，再传给 `CriticRequest`

### P0-6 ✅ 第二轮 GPU 交接（vLLM stop/start）

**问题：** vLLM 常驻到整个 `controller.run()` 结束，第二轮 Wan 生成时 vLLM 仍占约 16 GB 显存，存在 OOM 风险。

**修改：**
- `agents/wanphysics/run_loop.py`：新增 `_VllmHandoffCritic` 包装类；每轮 `evaluate()` 完成后若还有下一轮，调用 `_raw_critic.stop_vllm()` 先释放 GPU，下一轮生成完毕进入 evaluate 前 `start_vllm()` 自动重启

### P0-7 / P0-8 ✅ Trial 采集生产入口（轻量版）

**问题：** `JsonlTrialRecorder`、`LearningRepairLoopRunner` 只有框架，无生产入口；`actual_trial_count=0`。

**修改（轻量版 N1）：**
- `agents/wanphysics/run_loop.py`：`controller.run()` 结束后从 `result.history` 提取各轮 before/after physics_score、decision、detector_backend，写 `trials.jsonl`
- `agents/wanphysics/batch_run_loop.py`：同样写每个 prompt 的 `trials.jsonl`

### P1-1 ✅ Proxy Memory 加载并注入混合策略

**问题：** `repairer.py` 只加载 `TorchActionValuePolicy`，checkpoint 里的 `proxy_memory_train.jsonl` 未使用。

**修改：**
- `generators/wanphysics/repairer.py`：`load_action_value_repairer()` 检测 `memory/proxy_memory_train.jsonl`，加载 `RepairMemory`；`ActionValueRepairer.repair_with_decision()` 在有 matches 时以 `memory_weight=0.25` 混合 `action_distribution` 修正 per_action_values

### P1-2 ✅ 消除 Prompt Executor 双重 Policy 调用

**问题：** 若 `PromptRepairExecutor` 注入 `ActionValueRepairer`，会在 Runner 已有 `request.decision` 后再次运行 Policy 并递增 attempt 状态。

**修改：** `controller.py` 直接调 `repair_with_decision()` 拿决策后分发，`PromptRepairExecutor` 不再在 `LoopController` 主路径中被调用，绕过了二次决策问题

### P1-3 ✅ semantic/quality guardrail 接口预留

**修改：** `run_loop.py` summary 增加 `semantic_score: null`、`quality_score: null` 占位字段，接口对齐 `LearningRepairLoopRunner` 设计，待 scorer 接入后填充

### P1-4 / P1-5 ✅ detector_backend provenance 保全

**问题：** `eval_step.py` 写入 payload 的 `detector_backend` 在返回时被丢弃，`loop_result.json` 无法审计实际使用了哪个 Critic 后端。

**修改：**
- `generators/wanphysics/sam2_vlm_critic.py`：把 `detector_backend` 注入 `report.diagnostics["detector_backend"]`，随 report 全程传递；同步写入 `critic.json`
- `agents/wanphysics/run_loop.py`：summary 从 `result.best.report.diagnostics` 取出写入 `loop_result.json`

### P1-6 ✅ 最后一轮跳过 Repairer 调用

**问题：** `controller.py` 在最后一轮未 accepted 后仍调用 `repair_with_decision()`，产生"有决策无执行"的审计歧义。

**修改：**
- `src/physgenloop/controller.py`：`round_index == max_rounds - 1` 时直接 `break`，不触发 Repairer

### P1-7 ⏭ 绝对路径硬编码

sy-check 指出代码违反 README "禁止硬编码绝对路径"约束，`run_loop.py`、`eval_step.py`、`configs/loop.yaml` 中存在大量 `/root/PhysGenLoop-`。**本次不修改**，因为当前服务器路径固定，改为相对路径需要同步修改所有子进程调用链和 sys.path 策略，风险较高，留待后续统一处理。

### P1-8 ⏭ VideoSearch 模块缺失

sy-check 标注为"若 VideoSearch 是构想中需要参与的模块"——已确认该功能最初在计划中但未立项实现，**本次略过**。

## 二、htz_check.md 对齐结果

### 差异 1（critic.json）✅ 已符合

gen_step.py 已在生成阶段写 `status=waiting` 的 critic.json，`sam2_vlm_critic.py` 完成评估后回填 `status=completed`。本次追加写入 `detector_backend` 字段，进一步完善。

### 差异 2（纯生成入口）✅ 已符合

服务器已存在 `agents/wanphysics/generate_only.py`，符合 htz 要求。

### 差异 3（batch task_id 来自文件名）✅ 已符合

`batch_run_loop.py` 使用 `pf.stem` 作为 `prompt_id`，输出到 `outputs/{task_id}/`，符合原始约定。

### 差异 4（输出目录结构）✅ 部分符合

目录为 `outputs/run_时间/{candidate_id}/`，htz 期望 `outputs/{task_id}/`。batch 模式下 task_id 已来自文件名；run_loop.py 单 prompt 模式目录结构保持不变（属于 LoopController 架构决策，不是 bug）。

### 差异 5（critic.json 接口解耦）⏭ 架构层面

htz 期望生成器和 Critic 通过文件接口完全解耦。当前 `WanSubprocessGenerator` 通过 Python 对象接口与 `LoopController` 交互，critic.json 是副产物而非主接口。这是架构设计权衡，不做改动。

### 差异 6（candidate_id 可读格式）✅ 部分改善

htz 期望 `wan-{seed:04d}` 格式，当前用 `sha256[:12]`。本次**未修改** candidate_id 策略，因为 sha256 保证唯一性、避免并发冲突，是工程需要。可读性问题通过 `trials.jsonl` 和 `loop_result.json` 里的 seed 字段查询。
