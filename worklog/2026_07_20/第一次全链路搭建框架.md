## 一、当前全链路入口

当前批量跑 videophy2 prompt 的入口是：

```text
agents/wanphysics/run_videophy2_loop.py
```

它负责做四件事：

1. 从 manifest 或 CSV 读取样本
2. 为每个样本组装一套完整闭环
3. 逐条串行运行生成 → 评估 → 修复 → 再评估
4. 把单样本结果和整批汇总写到 `outputs/`

默认使用的 manifest 是：

```text
/root/PhysGenLoop-/evaluation/manifests/videophy2_pilot300.json
```

对 videophy2 这套 manifest，实际读取的 prompt 字段是：

```text
sample["prompt"]
```

## 二、整体执行顺序

当前全链路的真实运行顺序如下：

```text
videophy2 manifest / CSV
    ↓
run_videophy2_loop.py 读取 sample_id + prompt
    ↓
组装 LoopController
    ↓
LoopController 每轮执行：
    1. 生成前强制清理 vLLM / GPU 显存
    2. WanSubprocessGenerator 生成视频
    3. Sam2VlmSubprocessCritic 评估视频
    4. EvidenceAwareSelector 选当前轮最佳候选
    5. 达标则 accepted
    6. 不达标则 ActionValueRepairer + error_scope 选择修复动作
    7. 进入 prompt repair / global regeneration / local editing / reject
    8. 进入下一轮
    ↓
输出 loop_result.json / trials.jsonl / summary.json
```

## 三、核心编排层：LoopController

核心编排器在：

```text
src/physgenloop/controller.py
```

它是当前全链路的主控层，职责是：

1. 按轮次执行生成和评估
2. 在单轮内做 Best-of-K 候选选择
3. 检查是否达到 acceptance 条件
4. 调用 repairer 生成修复决策
5. 根据决策动作分发给不同执行路径

当前接受条件是：

```text
report.decision == "physical"
and
report.physics_score >= acceptance_score
```

如果满足，就直接返回：

```text
stop_reason = "accepted"
```

否则进入修复逻辑。

---

## 四、生成阶段：WanSubprocessGenerator

生成器在：

```text
generators/wanphysics/adapter.py
```

当前真正用于全链路的是：

```text
WanSubprocessGenerator
```

设计重点：

1. 不是在主进程常驻加载 Wan2.2
2. 每次 `generate()` 都单独起一个子进程执行 `agents/wanphysics/gen_step.py`
3. 子进程结束后 CUDA 上下文随进程退出一起释放

这一步解决的是：

```text
Wan2.2 和 vLLM 不能长期同时占同一张 40GB A100
```

生成输出包括：

- 视频文件
- `prompt.txt`
- `metadata.json`
- 初始化状态的 `critic.json`

候选目录结构是：

```text
outputs/.../{sample_id}/{candidate_id}/
```

---

## 五、评估阶段：Sam2VlmSubprocessCritic

评估器在：

```text
generators/wanphysics/sam2_vlm_critic.py
```

它负责：

1. 管理 vLLM 生命周期
2. 启动 `agents/wanphysics/eval_step.py`
3. 把候选视频送进 SAM2 + VLM Critic
4. 恢复 `CriticReport`
5. 回填 `critic.json`

### 5.1 显存串行化

这是当前全链路能跑通的关键。

现在新增了强制串行化机制：

- 生成前先调用 `prepare_for_generation()`
- 如果 vLLM 还活着，或者 GPU 显存还没降下来，就先执行 `stop_vllm()`
- `stop_vllm()` 不只杀 `api_server`，还会清：
  - `VLLM::EngineCore`
  - `vllm.entrypoints.openai.api_server`
  - `vllm`
- 然后轮询 `nvidia-smi`，等显存降到阈值以下再允许下一步生成

这一步把“串行交接显存”从约定变成了强约束。

### 5.2 eval_step.py 的职责

`agents/wanphysics/eval_step.py` 只做评分，不做 repair 决策。

它会：

1. 从 candidate JSON 恢复 `GeneratedCandidate`
2. 恢复 `PhysicsPlan`
3. 构造 `PhysicsCritic`
4. 优先尝试 `SAM2ObjectDetector + VLM`
5. 失败时降级到规则 Critic

输出字段至少包括：

- `decision`
- `physics_score`
- `violations`
- `diagnostics.detector_backend`

### 5.3 SAM2 mask 落盘

`src/pavg_critic/sam2_detector.py` 现在已经支持：

1. 跟踪阶段缓存每个对象、每个 frame 的二值 mask
2. 只对最终 `violation.critical_frames` 落盘 mask
3. 在 `eval_step.py` 里把 `mask_uri` 和 `mask_uris` 写回 `Violation.evidence`

这样后面的局部修复就不再是盲修，而是能拿到真正的局部区域。

## 六、选择阶段：EvidenceAwareSelector

候选选择器在：

```text
src/physgenloop/selector.py
```

全链路中实际注入的是：

```text
EvidenceAwareSelector
```

作用是：

1. 在单轮多个候选里选最佳
2. 在 `reject` 场景下，从已有历史最好候选里选最终返回对象

因此它既参与单轮 Best-of-K，也参与 `REJECT` 的兜底收口。

---

## 七、修复决策层：ActionValueRepairer

修复决策入口在：

```text
generators/wanphysics/repairer.py
```

它本质上是一个适配层，把：

```text
ActionValueDecisionPolicy
```

适配成 `LoopController` 可直接消费的 repairer。

### 7.1 当前职责

它负责：

1. 维护 `attempt_index`
2. 维护 `previous_actions`
3. 构造 `RepairContext`
4. 调用 policy 输出 `RepairDecision`
5. 根据 instruction 生成修复后 prompt

### 7.2 已实现的能力点

现在已经支持：

- `local_editor_available` 不再硬编码为 `False`
- 尝试加载 proxy memory
- 如果 policy 没给出 `local_target`，则根据 `violation` 自动补一个 target

当前仍有一个非阻塞告警：

```text
proxy_memory_train.jsonl 里 target 格式不符合当前契约
```

所以运行时会看到：

```text
failed to load proxy memory
```

但这不会阻断主链路。

## 八、错误范围判定：error_scope

错误范围启发式在：

```text
generators/wanphysics/error_scope.py
```

作用是对 policy 决策做一层可解释兜底。

当前规则：

1. 聚合所有 `violation.critical_frames`
2. 计算：

```text
ratio = len(unique_critical_frames) / total_frames
```

3. 判定：

- `ratio == 0` → `global`
- `ratio >= threshold` → `global`
- `0 < ratio < threshold` → `local`

在 `LoopController` 中的使用方式是：

- 若判定为 `local`，且有 `mask_uri/critical_frames`，并且执行器支持 `local_editing`，则强制走 `LOCAL_EDITING`
- 若判定为 `global`，则不允许错误地落到局部修复

并且最终会把轨迹写进：

```text
report.diagnostics["error_scope"]
```

便于后续审计。

## 九、执行器层：四动作分发

执行器注册在：

```text
generators/wanphysics/executor_factory.py
```

当前已经接入真实后端，构成四动作：

### 9.1 PROMPT_REPAIR

- 执行器：`PromptRepairExecutor`
- 后端：`ActionValueRepairer + WanSubprocessGenerator`
- 行为：改 prompt，然后全局重新生成

### 9.2 GLOBAL_REGENERATION

- 执行器：`GlobalRegenerationExecutor`
- 后端：`WanSubprocessGenerator`
- 行为：回到原始 prompt，重新生成

### 9.3 LOCAL_EDITING

- 执行器：`LocalEditingExecutor`
- 后端：`ProPainterLocalEditor`
- 行为：对 `mask_uri + critical_frames` 指向的局部区域做视频修复

### 9.4 REJECT

- 执行器：`RejectExecutor`
- 后端：`EvidenceAwareSelector`
- 行为：提前终止，并从已有候选中选择最终最优

## 十、局部修复层：ProPainterLocalEditor

局部编辑器在：

```text
generators/wanphysics/local_editor.py
```

输入是：

- `GeneratedCandidate`
- `LocalEditTarget`
- `instruction`

当前逻辑：

1. 拆视频为帧
2. 根据 `mask_uri` 读参考 mask
3. 用 `critical_frames` 决定哪些帧需要修
4. 调 ProPainter 子进程
5. 重编码输出视频

如果 `mask_uri` 缺失，会退化为全白 mask，也就是近似全帧修复。

## 十一、输出组织方式

批量运行 videophy2 的输出根目录是：

```text
/root/PhysGenLoop-/outputs/videophy2_run_时间戳/
```

结构如下：

```text
videophy2_run_xxx/
├── summary.json
├── vllm.log
└── {sample_id}/
    ├── loop_result.json
    ├── trials.jsonl
    └── {candidate_id}/
        ├── {candidate_id}-v01.mp4
        ├── prompt.txt
        ├── metadata.json
        ├── critic.json
        └── sam2_masks/
```

### 11.1 summary.json

整批汇总：

- 样本数
- accepted 数
- acceptance_rate
- average_rounds
- local_editing_count
- global_scope_count

### 11.2 loop_result.json

单样本汇总：

- `stop_reason`
- `best_candidate_id`
- `best_video_path`
- `best_physics_score`
- `best_decision`
- `detector_backend`
- `error_scope_trace`

### 11.3 trials.jsonl

单样本每轮轨迹：

- round_index
- before / after physics_score
- before_decision
- before_detector_backend
- stop_reason

## 十三、当前仍存在的已知问题

### 13.1 Proxy memory 加载告警

`proxy_memory_train.jsonl` 中 target 与当前契约不一致，导致 repairer 启动时有 warning。

影响：

- 不影响主闭环运行
- 会导致 memory mixing 暂时未生效

### 13.2 SAM2 的后处理警告

日志里有：

```text
cannot import name '_C' from 'sam2'
```

当前表现看不阻断主链路，但说明 SAM2 的某些后处理扩展没有完整启用。

### 13.3 局部修复分支尚未做针对性 smoke

虽然代码链路已经接通：

- `mask_uri`
- `LOCAL_EDITING`
- `ProPainterLocalEditor`

但还需要构造一个明确触发局部错误的样本，单独验证一轮：

```text
violation -> local scope -> local editing -> re-evaluate
```
